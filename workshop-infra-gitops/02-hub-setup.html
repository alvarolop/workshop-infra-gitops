2. Deploy Argo Hub Setup

2.1 Lab overview

In this workshop we are covering a GitOps infrastructure configuration approach. We will set an argo of argos and app-of-apps architecture to manage setup on destination clusters, and will apply day 2 operations on managed clusters.

Argo Hub is a common OpenShift cluster for every user in this workshop that will be used for deploying the basic infrastructure on managed clusters using ApplicationSet. Managed clusters are a set of Single Node OpenShift, alias SNO, available for every user and day 2 operations will setup based on ArgoCD instance running on managed cluster.

You will be able to connect to both Argo Hub and managed clusters.

Take a look at the general overview of the lab.

Figure 1. Laboratory overview

First of all, when you install GitOps operator, it will deploy a default argo instance which is intended for cluster configuration. So in this instance you will find an Application gitops-setup which is deploying and managing a second argocd instance and its configuration.

Figure 2. Argo Hub instances

As an user lab, you will access the second argo instance where you will have privileges enough thanks to RBAC configuration to deploy an Application for applying ApplicationSet to apply configuration on your managed cluster. In this instance you can expect two ApplicationSet: gitops-setup and bootstrap-sno.

Figure 3. Argo Hub ApplicationSet

gitops-setup ApplicationSet will install the openshift-gitops operator and the argocd-infra instance, while the second one will generate 'N' Applications on managed clusters for bootrstrapping the argocd-apps instance and setup its configuration.

Figure 4. Argo Hub deploying on SNO

Once GitOps is deployed on your managed cluster you can start configuring day 2 operations locally in your new provisioned argocd-infra instace.

Figure 5. Day 2 operations and applications deploy

2.2 Environment

You have access to two clusters: Argo Hub and your own SNO:

  • The Argo Hub is a normal OpenShift cluster that will be used to control the full fleet of clusters. It will be shared by all the participants of the workshop.

  • The SNO, or Single Node OpenShift, is a smaller cluster that will be individual to each of you so that you can experiment freely. Learn more about SNO, our managed cluster, in this workshop.

Replace XX values of the OCP_USER, OCP_SNO_API, and OCP_SNO_CONSOLE with the values provided by the instructor at the beginning of this workshop. After that, keep those variables at hand and present every time that you execute commands in your terminal.

OCP_USER=userXX
OCP_PASS=redhat
DOMAIN=sandbox2414.opentlc.com
OCP_HUB_API=https://api.argo-hub.sandbox2414.opentlc.com:6443
OCP_HUB_CONSOLE=https://console-openshift-console.apps.argo-hub.sandbox2414.opentlc.com
OCP_SNO_API=https://api.sno-XX.sandbox2414.opentlc.com:6443
OCP_SNO_CONSOLE=https://console-openshift-console.apps.sno-XX.sandbox2414.opentlc.com

Make sure to use the correct values for OCP_USER, OCP_PASS, OCP_HUB_API, OCP_HUB_CONSOLE, OCP_SNO_API, and OCP_SNO_CONSOLE as provided.

2.3 Accessing the Argo Hub

Let’s start logging in to Argo Hub cluster API:

oc login -u $OCP_USER -p $OCP_PASS $OCP_HUB_API

You should also log in trough the web console. This way you will be able to jump between Argo Hub and SNO clusters easily.

Change <domain> information, provided by the instructor at the beginning of this workshop.

  • Open $OCP_HUB_CONSOLE (E.g. https://console-openshift-console.apps.argo-hub.<domain>/)

  • Hit Workshop Users.

Figure 6. Argo Hub log in

  • Insert userXX and <pass> provided by instructor and click Log in button.

Figure 7. Insert Argo Hub credentials to log in

On top of that, once logged into Argo Hub you will only have permissions over your project and destination cluster.

The reason why Argo Hub has permissions over your managed cluster is because it is configured as a destination server, given an authentication method.

In Argo Hub there is a secret in openshift-operators namespace for every managed cluster containing this information. There are several ways to implement this, you can generate an authentication token for this purpose or you can pass kubeconfig file to give full control over the destination cluster. You do not need to do anything as this is already done on hub-setup Application.

Figure 8. Managed cluster secret in Argo Hub

You can take a look to the chart used for creating these secrets here and here.

2.4 Overview of the Hub Setup Application

In the Argo Hub cluster you can find two argo instances in different namespaces:

  • openshift-gitops (default) in the openshift-gitops namespace. The default instance is intended to manage the hub itself, and it is in charge of deploying the argocd instance and its configuration.

  • argocd in the openshift-operators namespace which is dedicated for managed clusters setup and configuration.

Argo Hub setup should be done only once before this workshop, you can log in to openshift-gitops instance console.

Change <domain> information, provided by the instructor at the beginning of this workshop.

  • Open <openshift-gitops-instance-console> (E.g. https://openshift-gitops-server-openshift-gitops.apps.argo-hub.<domain>)

  • Click LOG IN VIA OPENSHIFT.

Figure 9. Log in openshift-gitops console instance

  • Hit Workshop Users.

Figure 10. OpenShift credentials for openshift-gitops instance

  • Insert userXX and <pass> provided by instructor and click Log in button.

Figure 11. openshift-gitops instance log in

This user has view permissions in openshift-gitops instance. You can take a look to the objects that ArgoCD is managing for deploying the second instance.

Figure 12. hub-setup overview

Then you can go to the code to take a look to the helm charts used.

As this configuration is already deployed you do not need to make any change, only observe the way it was done.

  • Go to your workshop-gitops-content-deploy repository, navigate to the hub-setup Application code and see how this sets up env configuration on in-cluster (https://kubernetes.default.svc):

cd ~/workshop-gitops-content-deploy
git checkout setup-sno
cat global-config/bootstrap-a/hub-setup-a.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: hub-setup
  namespace: openshift-gitops
spec:
  destination:
    namespace: openshift-gitops
    server: https://kubernetes.default.svc
  project: default
  source:
    repoURL: https://github.com/alvarolop/workshop-gitops-content-deploy.git
    targetRevision: setup-sno
    path: hub-setup/charts/gitops-setup
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

Check the openshift-gitops console and take a look to the objects managed by argo in the hub-setup Application. As you can see, using range of values while rendering charts allows to create charts for multiple destinations without much complexity.

Figure 13. hub-setup Application in openshift-gitops instance console

For this initial setup it is only required to create argocd instance, App Projects, Groups, Role Bindings and Secrets for cluster destinations.

2.5 Accessing the Managed Cluster

Once argocd instance (The second) is up and running, you can login to the ArgoCD console with your userXX:

Change <domain> information, provided by the instructor at the beginning of this workshop.

  • Open <argocd-instance-console> (E.g. https://argocd-server-openshift-operators.apps.argo-hub.<domain>)

  • Click LOG IN VIA OPENSHIFT.

Figure 14. Log in argocd console instance

  • Hit Workshop Users.

Figure 15. OpenShift credentials for argocd instance

  • Insert user admin and redhat provided by instructor and click Log in button.

Figure 16. ArgoCD instance log in

In the argocd instance, your user will have admin permissions for your own applications, while you will only have cluster reader permissions when you login to OpenShift Argo Hub console.

By using argocd instance you will be able to setup your own managed cluster.

Ideally you could set up every managed with a single ApplicationSet by configuring multiple cluster-definition. However because of the workshop structure you need to perform cluster scoped configurations without affecting other users, but your repository could be easily adapted to perform actions on multiple clusters.
